01 · Background

Problem statement

Knowledge cutoff

Training data ends at a fixed date. Private and recent documents are not part of the model.

Context limits

Full documents do not fit in a prompt. Relevant passages must be selected automatically.

Hallucination

Beyond its knowledge, a model generates fluent but unsupported text. Without sources, errors are undetectable.

RAG — Retrieval-Augmented Generation — addresses all three: retrieve relevant passages first, then constrain the model to answer only from them.

02 · Ingestion

Two pipelines: ingestion & query

INGEST — once per document

Offline, in the worker. Can afford to be slow and thorough.

# apps/worker — arq jobs parse_source PDF → clean page text embed_source chunk → vectors → index outline_source suggested questions

QUERY — every question

Live, user waiting. Every millisecond and token counts.

# packages/core — LangGraph rewrite follow-up → standalone query retrieve hybrid RRF → rerank → top 8 generate Claude + citations, streamed map_citations offsets → source spans

Answer quality is determined at ingestion time; response latency at query time. The two are engineered separately.

02 · Ingestion — step 1

Step 1 — parsing & normalization

extractPyMuPDF reads each page in natural order → one source_part per page, char offsets preserved.
strip boilerplateEdge lines repeating on >60% of pages — running titles, "Page 3" — are chrome, not content. They'd pollute every chunk's embedding.
de-hyphenateimprove-\nment → improvement — split words hide from both keyword and vector search.
detect structureFont metadata reveals headings: short line, noticeably larger than body text → a heading, with its level and char offset. (new)

Every downstream stage inherits parsing quality — normalization is performed once, at ingestion.

02 · Ingestion — step 1½

Heading detection

# body size = the font size carrying # the most characters in the document body = weights.most_common(1) # heading = short line, ≥12% larger # (or bold at body size) if len(line) ≤ 100 and size ≥ body × 1.12: heading(level=rank_of(size), start=char_offset)
no ML, no depsPure font statistics — zero new dependencies, CPU-only, runs at parse speed.
levels for freeDistinct heading sizes rank into levels: biggest font → level 1.
offsets matterEach heading is located in the cleaned text — its char offset becomes a hard chunk boundary next step.
02 · Ingestion — step 2

Step 2 — structure-aware chunking

1Paragraphs pack

Blank-line blocks pack greedily to ~1200 chars (≈300 tokens). Chunks start and end on paragraph edges.

2Sentences split

A paragraph too big to fit splits at sentence ends — never mid-thought. Whitespace windows only as last resort.

3Headings cut hard

No chunk ever spans a section boundary — a retrieval hit never mixes two sections.

The citation contract — the invariant everything hangs on

source_text[chunk.char_start : chunk.char_end] == chunk.text

Every chunk is an exact slice of its page. Break this, and citations point at the wrong words. 1,300 randomized tests enforce it.

Orphan chunks carry document + section identity into vector space — without touching the cited text.

02 · Ingestion — step 4

Step 4 — storage & indexing

-- the chunk table, heart of retrieval text the raw slice (citation contract) embedding vector(384) → HNSW index, cosine -- dense / semantic tsv tsvector → GIN index, weighted -- sparse / keyword meta {page, source_title, section_path} char_start, char_end -- offsets into the source page
why both?Vectors catch paraphrase; keywords catch exact terms — part numbers, names, acronyms that embeddings fumble.
why weighted?Body text indexes at weight A, title + section path at B — a heading keyword helps, but a body match wins.

03 · Retrieval — node 1

Query rewriting

Problem

turn 1: "what were the three findings?" turn 2: "tell me more about the second one" searching for "the second one" retrieves… nothing useful.

Approach

cheap model + history → "second finding: reranking improves precision at low k" now retrieval has something to bite.

Best-effort by design: turn 1 passes through untouched, and any failure falls back to the raw question — the rewrite may improve a turn, never break one.

03 · Retrieval — node 2

Hybrid retrieval with RRF

-- one CTE, one round-trip dense : ORDER BY embedding <=> query_vec -- pgvector cosine, top 50 sparse : WHERE tsv @@ websearch_to_tsquery(query) -- keyword match, top 50 fused : SUM(1.0 / (60 + rank)) -- Reciprocal Rank Fusion
why RRF?Similarity scores live on incomparable scales — ranks compare. High on either list → surfaces.
why 60?Standard damping — softens rank 1 vs rank 10, so neither list dominates.
scopedEvery arm filters notebook_id first — tenant isolation lives in the query.

04 · Generation

Grounded generation

search_result blocksThe 8 chunks reach Claude as native content — the Citations API marks which characters support each sentence.
grounding prompt"Answer using ONLY the provided search results… if the answer isn't there, say so."
prompt cachingBreakpoints cache the stable prefix across turns; only fresh results + the new question pay full price.
history trimmingThe model sees a 12-message tail; full history stays in checkpoints for the UI.

Insufficient evidence → the system reports "not found in the documents."

06 · Analysis

Evaluation methodology

Citation grounding

The headline gate. Fraction of citations whose char-offset roundtrip resolves exactly. Ship requires ≥ 0.90.

Lite metrics

Word-overlap proxies for faithfulness & relevancy — no LLM calls, instant feedback while tuning top_k.

Real Ragas

LLM-judge scoring for depth, imported lazily — opt-in, so the base eval stays light and cheap.

Known limitation

The golden set is 10 questions. Fine for a smoke test; far too small to catch regressions reliably. Growing it to 50–100 labeled questions is the single cheapest quality investment in the roadmap.

06 · Analysis

Comparative analysis

Implemented

▸ Retrieval funnel — hybrid, fused, reranked
▸ Database layer — vectors + keywords, one store
▸ Verified citations — beyond most reference designs
▸ Eval harness with a hard ship gate
▸ Structure-aware ingestion (shipped this week)

Not implemented

▸ Reasoning engine — no routing, planning, or retry
▸ Human feedback loop — no 👍/👎, nothing learns
▸ Red teaming — PDF prompt-injection untested
▸ Data sources — PDFs only; no OCR, tables, web
▸ Continuous eval — pre-ship only, not on live traffic

Summary: the core pipeline is complete and tested; the surrounding control loops remain future work.

06 · Analysis — case study

Case study: structure-aware chunking

Before

Blind 1200-char windows, whitespace-snapped.
Cut mid-sentence, mid-topic, across section borders.
Undersized tails silently dropped.

After

Paragraph packing → sentence splits → heading hard-cuts.
Section paths in chunk metadata and embeddings.
Tails merge; nothing lost; contract intact.

methodfont sizes → headings → boundaries → chunks → section paths → "title > 3 Methods > 3.2 Training" in the vector
validation13 new tests, randomized contract checks with random boundaries, and a schema migration — existing sources are re-ingested to benefit.
